\section*{1. Introduction}
Financial market interfaces often compress a complex matching process into a
simple price chart. For a casual trader, it is easy to think that if a stock is
currently shown at a certain price, then any order can be filled immediately at
that exact level. In reality, that is only true when there is enough available
liquidity at the relevant price levels and when the order can move through the
queue with acceptable slippage. The gap between a visible market price and an
actual fill price is one of the most important practical ideas in market
microstructure, and it is also a good setting for applying the tree-based data
structures and simulation techniques from CSC111.

Our project studies this gap by simulating a limit order book in pure Python. We
model bids and asks as separate binary search trees of price levels, maintain
FIFO priority within each level, replay event streams through a matching engine,
and compute summary statistics about fills, cancellations, spread, midpoint,
and slippage. On top of the simulator, we added a lightweight machine learning
component that observes book-state features and classifies a short-horizon
market action.

\textbf{Our project goal is to build a tree-based market simulator that can
replay realistic order flow, expose the mechanics of price-time-priority
matching, and train a small neural network to classify short-horizon market
actions as buy, sell, or hold from order-book state.}

This final submission keeps the core direction from our proposal, but the final
product is more grounded in a robust command-line simulation and training
workflow than in a polished browser interface. That shift let us focus on the
central computational challenge of the project: representing and updating a live
order book efficiently with trees while connecting that simulation to a
meaningful data and model-training pipeline.

\section*{2. Datasets}
Quantyze uses two kinds of datasets during development and one packaged dataset
zip for grading.

\subsection*{2.1 Synthetic Event Data}
We generated our own synthetic event streams for testing and demonstration. These
are created inside \texttt{data\_loader.py} and include three scenarios:
\begin{itemize}
    \item \texttt{balanced}
    \item \texttt{low\_liquidity}
    \item \texttt{high\_volatility}
\end{itemize}

These synthetic datasets use the same canonical event format as the rest of the
project: \texttt{timestamp}, \texttt{order\_id}, \texttt{side},
\texttt{order\_type}, \texttt{price}, and \texttt{quantity}. The synthetic data
is useful because it gives us a deterministic way to demonstrate fills,
cancellations, and changing book state without requiring a large external
download.

\subsection*{2.2 LOBSTER Sample Data}
For the real-data path of the machine learning portion of the project, we used
LOBSTER sample order-book data from the official LOBSTER sample page.\footnote{\url{https://data.lobsterdata.com/info/DataSamples.php}}
LOBSTER provides two paired files:
\begin{itemize}
    \item a message file containing timestamped market events
    \item an order-book file containing depth snapshots
\end{itemize}

Our code supports raw LOBSTER message and order-book CSVs. From the message
file, we use normalized information corresponding to timestamp, event type,
order id, size, price, and direction. The loader also recognizes message types
that should be preserved as annotations rather than replayed directly.

From the paired order-book file, our current feature pipeline uses only the top
two bid levels and top two ask levels, even when a file contains deeper book
data. This means the model ultimately trains on level-1 and level-2 price/size
information rather than the full book.

We used public LOBSTER sample files such as AAPL, AMZN, and GOOG during
development to validate the real-data pipeline. However, for the final shipped
saved state in this submission, we use packaged internal-format data so that the
TA can reproduce the workflow quickly without depending on external market-data
downloads.

\subsection*{2.3 Submission Dataset Package}
For the final submission, the dataset zip is \texttt{quantyze\_datasets.zip}. It
contains:
\begin{itemize}
    \item \texttt{sample\_internal.csv}
    \item \texttt{huge\_internal.csv}
    \item \texttt{instructions.txt}
    \item \texttt{model.pt}
    \item \texttt{training\_metrics.json}
\end{itemize}

The small sample dataset is meant for a short under-one-minute training check.
The larger internal dataset is a baseline-scale retraining demo. The packaged
saved-state workflow also includes a shipped baseline checkpoint and metrics so
that the TA can use the model path without retraining first.

\section*{3. Computational Overview}
\subsection*{3.1 Core Data Representation}
The central data structure in Quantyze is the limit order book. We represent the
book with two binary search trees:
\begin{itemize}
    \item one tree for bids
    \item one tree for asks
\end{itemize}

This logic is implemented across \texttt{book\_tree.py},
\texttt{price\_level.py}, and \texttt{order\_book.py}. Each node in a tree is a
\texttt{PriceLevel} object. A price level stores:
\begin{itemize}
    \item a price
    \item aggregate resting volume at that price
    \item a FIFO queue of resting orders
\end{itemize}

The trees let us preserve sorted price structure while still supporting
efficient best-bid and best-ask access. The FIFO queue inside each price level
enforces time priority among orders resting at the same price. This combination
directly models price-time-priority matching, which was the main algorithmic
goal of our project.

\subsection*{3.2 Event Replay and Matching}
All incoming data is converted into a common \texttt{Event} representation in
\texttt{orders.py}. The \texttt{DataLoader} in \texttt{data\_loader.py} either:
\begin{itemize}
    \item loads a dataset from internal CSV or raw LOBSTER files, or
    \item generates a synthetic scenario
\end{itemize}

Those events are then replayed through \texttt{EventStream} in
\texttt{event\_stream.py}, which emits them one at a time into the
\texttt{MatchingEngine} in \texttt{matching\_engine.py}. The matching engine
handles limit orders, market orders, and cancellations. It updates the
\texttt{OrderBook}, performs partial and full fills, removes empty price levels
when necessary, and records execution information used for later metrics.

At the end of a normal run, \texttt{main.py} prints a terminal summary with:
\begin{itemize}
    \item total filled quantity
    \item fill count
    \item cancel count
    \item average slippage
    \item final spread
    \item final midpoint
    \item current mark-to-market agent P\&L if a trained agent is active
\end{itemize}

The current default run path is command-line based rather than browser based.
This means the project reports results through terminal output, the packaged
saved-state artifacts extracted from the dataset zip, and runtime logs such as:
\begin{itemize}
    \item \texttt{model.pt}
    \item \texttt{training\_metrics.json}
    \item \texttt{log.json}
\end{itemize}

\subsection*{3.3 Data Processing and Training Pipeline}
The machine learning pipeline is built mainly in \texttt{data\_loader.py},
\texttt{neural\_net.py}, and \texttt{main.py}.

The current feature pipeline uses 16 features per example:
\begin{enumerate}
    \item best bid price
    \item best bid size
    \item best ask price
    \item best ask size
    \item spread
    \item mid-price
    \item level-1 imbalance
    \item second bid price
    \item second bid size
    \item second ask price
    \item second ask size
    \item event-side feature
    \item one-step best bid delta
    \item one-step best ask delta
    \item one-step mid-price delta
    \item one-step imbalance delta
\end{enumerate}

The label policy is horizon-based. For each example, we compare the current
midpoint to the midpoint 50 events later:
\begin{itemize}
    \item if the future midpoint is more than \$0.01 higher, the label is
    \texttt{buy}
    \item if it is more than \$0.01 lower, the label is \texttt{sell}
    \item otherwise the label is \texttt{hold}
\end{itemize}

This pipeline is implemented in \texttt{DataLoader.build\_training\_dataset} and
related helper methods. The final training flow in \texttt{main.py} computes
train/validation splits, standardizes features with training-set mean and
standard deviation, applies class-weighted cross-entropy, trains a PyTorch
feed-forward network, and saves the latest training artifacts:
\begin{itemize}
    \item \texttt{latest\_model.pt}
    \item \texttt{latest\_training\_metrics.json}
    \item \texttt{latest\_training\_data.csv}
\end{itemize}

The shipped baseline saved-state artifacts remain \texttt{model.pt} and
\texttt{training\_metrics.json}, so the TA can always return to a known working
checkpoint.

\subsection*{3.4 User Interactivity}
The TA-facing interface is a nested terminal menu implemented in
\texttt{cli\_menu.py}. Running \texttt{main.py} with no arguments opens a menu
with separate sections for:
\begin{itemize}
    \item simulation
    \item training
    \item artifacts and metrics
    \item help / command reference
\end{itemize}

This menu lets the TA choose scenarios, replay datasets, run short training,
inspect baseline versus latest metrics, inspect artifact status, inspect the
current execution log, inspect the packaged dataset zip, and switch the active
simulation model between the shipped baseline checkpoint, the latest retrained
checkpoint, and no model. The active choice is stored in
\texttt{active\_model.json}, which makes the training-to-simulation workflow
persistent across runs.

\subsection*{3.5 Overall Computational Complexity}
Quantyze demonstrates computational complexity in all three major ways
highlighted by the project rubric.

\begin{itemize}
    \item \textbf{Non-trivial dataset processing and model training:} the system
    supports multiple input formats, infers paired LOBSTER order-book files,
    separates replay-compatible and non-replayable events, builds a shared
    training dataset, normalizes features, computes class weights, and saves
    checkpoints and metric artifacts.
    \item \textbf{Non-trivial algorithms implemented by us:} the BST-based order
    book, FIFO price-level queues, price-time-priority matching engine, and
    cancellation/indexing logic are all custom project code rather than thin
    wrappers around third-party libraries.
    \item \textbf{Non-trivial reporting and interactivity:} the nested terminal
    menu, saved-state workflow, metrics inspection, and execution-log reporting
    make the program substantially more interactive than a single fixed command.
\end{itemize}

\section*{4. Instructions for Obtaining Datasets and Running the Program}
Our TA-facing run path is designed as a workflow rather than as a collection of
independent commands. The key distinction is that \textbf{simulation source}
and \textbf{model source} are separate ideas:
\begin{itemize}
    \item the \textbf{simulation source} is the event stream being replayed,
    such as \texttt{balanced}, \texttt{low\_liquidity},
    \texttt{high\_volatility}, or a custom CSV
    \item the \textbf{model source} is the checkpoint currently selected for
    inference, such as the shipped \texttt{model.pt} baseline or a newly
    trained \texttt{latest\_model.pt}
\end{itemize}

This separation is intentional. It lets the TA test the same model under
different market conditions and also lets the TA retrain a model without
overwriting the shipped baseline artifacts.

\subsection*{4.1 Installing Libraries}
Install the libraries listed in \texttt{requirements.txt} before running the
program.

\subsection*{4.2 Obtaining Datasets}
Before using the packaged datasets, extract \texttt{quantyze\_datasets.zip}
beside \texttt{main.py}. This provides the small verification dataset, the
larger saved-state dataset, and the baseline saved artifacts.

\subsection*{4.3 Workflow Overview}
The primary grading path is the interactive menu, which answers four questions
at every stage:
\begin{itemize}
    \item What event source is being replayed?
    \item Which checkpoint, if any, is active?
    \item Where did that checkpoint come from?
    \item What is the next logical action: inspect, train, or simulate?
\end{itemize}

\subsection*{4.4 Recommended TA Demo}
The following walkthrough is the clearest way to understand the final
submission.

To use this path, the TA should run \texttt{main.py} with no additional
arguments. This opens the nested interactive menu.

We recommend the following sequence:
\begin{enumerate}
    \item Open \texttt{Artifacts \& Metrics} and choose \texttt{View active model status}. This
    confirms the currently active mode (\texttt{baseline}, \texttt{latest}, or
    \texttt{none}), the checkpoint path, and the model provenance label.
    \item Open \texttt{Simulation} and run \texttt{Run default synthetic demo
    (balanced)}. This is the cleanest first demonstration because it produces a
    live two-sided book with real fills, a defined spread, and a defined
    midpoint at the end of the run.
    \item Return to \texttt{Simulation} and run \texttt{low\_liquidity} and
    \texttt{high\_volatility}. These two scenarios demonstrate that the same
    active model can be evaluated under different event streams without
    changing the core matching engine.
    \item Open \texttt{Training}. For a short demonstration, use
    \texttt{sample\_internal.csv}. For a baseline-scale demonstration, use
    \texttt{huge\_internal.csv}. After training, the menu prints the latest
    output file paths and asks whether the new checkpoint should become the
    active simulation model.
    \item If the TA answers yes, open \texttt{Artifacts \& Metrics} and check
    \texttt{View active model status} again. The provenance should now point to
    the newly trained checkpoint rather than the shipped baseline.
    \item Re-run the \texttt{balanced} synthetic demo. This shows the exact same
    event stream as before, but now under a different active model. This is the
    simplest way to isolate the difference between \textit{market replay}
    behavior and \textit{agent overlay} behavior.
\end{enumerate}

\subsection*{4.5 How the TA Should Interpret the Output}
The terminal output contains two different classes of information, and keeping
them separate is the most important part of understanding Quantyze.

\subsubsection*{Simulation Run Configuration}
Before each run, the program prints:
\begin{itemize}
    \item the \textbf{simulation source}, such as \texttt{synthetic balanced}
    or a custom replay path
    \item the \textbf{active model mode}, such as \texttt{baseline},
    \texttt{latest}, or \texttt{none}
    \item the \textbf{active model path}
    \item the \textbf{model provenance}, such as the packaged baseline
    checkpoint or a newly trained dataset like \texttt{sample\_internal.csv}
\end{itemize}

This header tells the TA which market is being replayed and which checkpoint is
observing it. These are intentionally not the same thing.

\subsubsection*{Matching-Engine Metrics}
The following summary values describe the replay itself:
\begin{itemize}
    \item \texttt{Total Filled}: total executed quantity produced by the
    matching engine
    \item \texttt{Fill Count}: number of executions
    \item \texttt{Cancel Count}: number of cancellations processed
    \item \texttt{Average Slippage}: average execution-price deviation under the
    current slippage calculation
    \item \texttt{Spread}: final bid-ask spread
    \item \texttt{Mid Price}: final midpoint
\end{itemize}

These values are properties of the event stream and the matching engine. If the
TA replays the same scenario with \texttt{baseline}, \texttt{latest}, and
\texttt{none}, these market metrics should stay the same because the underlying
event stream is unchanged.

\subsubsection*{Agent Overlay Metric}
If an active checkpoint is loaded, the program also prints
\texttt{Current Mark-to-Market P\&L}. This value should be read carefully.

It is \textbf{not} a claim that the model is a profitable trading strategy. In
our current system, the neural network is trained to classify whether the
midpoint will move up, down, or stay roughly flat 50 events later. During
simulation, this prediction is converted into a simple paper-trading overlay
that increments or decrements inventory by one unit whenever the simulator
observes fills. Therefore:
\begin{itemize}
    \item a strong classification accuracy does not automatically imply a strong
    trading P\&L
    \item a negative mark-to-market value can reflect inventory accumulation in
    a thin market rather than a broken simulator
    \item the ML component should be interpreted as a proof-of-concept
    prediction overlay, not as a finished trading strategy
\end{itemize}

For example, \texttt{low\_liquidity} often ends with a wider spread and fewer
fills than \texttt{balanced}. A model can therefore finish with a non-zero
inventory and a negative mark-to-market value even when the matching engine
itself is behaving correctly.

\subsubsection*{Training Metrics}
The training menu prints validation accuracy, majority-class baseline,
per-class recall, and the confusion matrix. These values measure how well the
classifier reproduces the label rule derived from future midpoint movement.
They do not directly measure whether the simulated agent will earn positive
P\&L on every synthetic scenario.

\subsection*{4.6 Optional Raw LOBSTER Check}
If the TA wants to test the raw LOBSTER path, they can use the custom dataset
option from the interactive menu and provide a LOBSTER message CSV path. The
paired order-book path is inferred from the message filename by replacing
\texttt{\_message\_} with \texttt{\_orderbook\_}.

\section*{5. Changes from Proposal to Final Submission}
Several parts of the final project changed from our proposal.

First, the simulator core became stronger and more complete than the proposal
version. The final system now has a clearer separation between loading data,
replaying events, matching orders, computing summaries, and handling the
interactive menu. The binary-search-tree order book and FIFO price-level logic
remained central from the start, but the final project is much more concrete in
how those structures interact.

Second, our machine learning plan changed. In the proposal, we described either
a decision-tree or neural-network style predictor in a relatively open-ended
way. In the final submission, we committed to a PyTorch neural network and built
a full training pipeline with explicit features, normalization, class weighting,
saved checkpoints, and validation metrics.

Third, our display plan became more modest. The proposal emphasized a
React/Plotly-style interface. In the final code, the most polished and reliable
user-facing path is the command-line simulation, the nested terminal menu, and
the generated training artifacts. We chose to prioritize a robust tree-based
simulator and reproducible training path over a more ambitious UI layer.

Finally, our data plan became more realistic. Instead of relying only on
synthetic data or describing external data abstractly, the final project
supports both generated scenarios and raw LOBSTER files, while the saved-state
submission workflow is based on packaged internal data so the TA does not need a
large external retraining step.

\section*{6. Discussion}
Our project goal was to model the mechanics of a limit order book using trees
and to connect that simulation to a simple market-action classifier. The final
result does achieve that goal, even though the machine learning results are more
modest on real data than the simulator itself.

On the simulation side, the output is meaningful and easy to interpret. In the
current default run, the simulator produces active matching instead of a dead
book. The default \texttt{balanced} run reports 1750.0 total filled volume, 378
fills, 8 cancellations, a final spread of approximately 0.6, and a final
midpoint of 100.0. These values show that our event sequence is not merely
inserting passive orders forever; it is actually generating executions,
consuming liquidity, and rebuilding the book. That was important because one of
the main reasons for doing this project was to expose the difference between a
quoted market price and the mechanics required to fill an order.

We also think the project succeeds as a data-structures project. The binary
search trees are not decorative: they are the central representation of the
market. The need to maintain sorted price levels, extract best bid and best ask,
remove empty levels, and preserve FIFO priority within a level all fit the
tree-based design naturally. This gives the project genuine computational
substance beyond simple data loading or basic statistics.

The machine learning component works, but its results should be interpreted
carefully. The currently packaged baseline checkpoint and its accompanying
metrics artifact show a validation accuracy of approximately 0.99992 compared to
a majority-class baseline of approximately 0.33753. These numbers are useful
for verifying that the training pipeline, checkpoint loader, and saved-state
workflow all function correctly, but they are not a realistic claim about
real-market prediction.

The run-time P\&L shown in the simulator should also be interpreted as an agent
overlay metric rather than as a proof of profitability. The neural network is
trained on future midpoint labels, not on a direct trading reward, and the
simulation overlay uses those labels to build a simple paper position over time.
As a result, it is possible for the classifier to work as designed while still
producing a negative mark-to-market value in a thin or volatile synthetic
scenario.

On public LOBSTER sample data during development, the task was much harder and
performance was only modestly above a naive baseline. We therefore do not claim
that the current model is a high-quality trading strategy or a strong predictive
system for real markets. Its role in the project is best understood as a
lightweight proof of concept that connects book-state features to model output
inside a complete simulator.

This outcome still helps answer our project goal in a useful way. The goal was
not only to maximize predictive accuracy, but to build a complete computational
pipeline from market events to order-book state to derived features to model
output. In that sense, the ML side of the project is successful: it uses both
synthetic and real-data paths, produces saved checkpoints, supports inference
inside the simulator, and can be retrained from the command line or through the
interactive menu.

We encountered several limitations. The first was data realism versus
convenience. Synthetic data is easy to test and reason about, but it is not
enough to justify a strong predictive claim. Real LOBSTER data is much richer,
but it creates issues of preprocessing, class balance, and training difficulty.
A second limitation was model simplicity. We used a small feed-forward network
over handcrafted features rather than a stronger temporal model. This was a
reasonable CSC111 project choice, but it limits what the model can learn about
market dynamics. A third limitation was presentation scope. Our proposal imagined
a more polished UI layer, but most of our time ended up being invested in the
simulation engine, dataset handling, and ML pipeline.

If we were to continue this project, the best next steps would be to improve the
visualization layer, add a stronger sequence-aware model, and evaluate the
trained agent with more realistic trading metrics over multiple market days.
Even with those limitations, we believe the current submission succeeds as a
CSC111 project because it combines tree-based design, non-trivial event
processing, model training, and interpretable output in one coherent system.
